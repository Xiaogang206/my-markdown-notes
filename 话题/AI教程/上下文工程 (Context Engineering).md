上下文工程 (Context Engineering)

在“Agent 核心原理”的背景下，来源将**上下文工程（Context Engineering）** 视为构建高性能智能体的核心支柱。它不涉及改变模型本身的结构（训练），而是通过**精细化设计模型的输入内容**，让模型在有限的上下文窗口（Context Window）内表现得更准、更好、且成本更低。

以下是来源对上下文工程在 Agent 体系中作用及其核心技术的深度看法：

### 1. 核心定位：Agent 效果的“天花板”

来源认为，Agent 本质上是“大模型 + 工具”的组合，旨在感知并影响外界。

- **Agent 性能的关键**：如果模型的回答不令人满意，往往不是因为模型不够强，而是因为没有提供**足够清晰和完整的信息**，这正是上下文工程要解决的问题。
- **应对 Agent 的动态冗余**：Agent 的运行（如 React 模式）涉及多次“思考-行动-观察”循环，产生的工具执行结果会迅速填满上下文窗口。若管理不当，Agent 会因“记忆过载”而变慢变傻。

### 2. 上下文工程的四大核心策略

来源借鉴了相关的技术框架，将上下文工程归纳为四个维度：

- **保存上下文（Save）**：通过筛选和总结，将重要信息持久化存储（如内存或硬盘），在需要时重新发给模型（如 ChatGPT 的记忆功能）。
- **选择上下文（Select）**：从海量信息中精选关联度最高的内容。
  - **静态选择**：将必须遵守的规则（如 `cursorrules` 或 `cloud.md`）硬性放入每一条请求。
  - **动态选择**：利用 **RAG（检索增强生成）**等技术，根据用户问题动态提取最相关的片段或工具说明。
- **压缩上下文（Compact）**：对历史消息进行总结并丢弃原始冗余数据。例如 **Claude Code** 在窗口使用量超过 95% 时会自动运行 `auto-compact` 流程来降低 Token 消耗。
- **隔离上下文（Isolate）**：这是多智能体（Multi-Agent）协作的核心策略。通过 **SubAgent** 拥有完全独立的上下文窗口，处理复杂中间过程，仅向主智能体汇报最终结果，从而保持主对话的简洁高效。

### 3. “渐进式披露”：上下文管理的工程智慧

在 **Agent Skill** 的协议设计中，上下文工程体现为“渐进式披露”机制，极大地优化了 Token 经济学：

- **按需加载**：最初只向模型展示技能的元数据（名称和描述）；只有当模型识别出需要该技能时，才会加载详细的指令层。
- **深层资源按需触发**：更长的参考资料（Reference）仅在提到特定触发词（如“预算”）时才会被读取并加入上下文。
- **脚本执行的“零消耗”**：Skill 允许运行 Python 脚本，代码只被执行而不被模型阅读。这意味着即便是万行复杂的逻辑，占用的上下文 Token 几乎为零。

### 4. 行业趋势：从转圈调度到端到端命令

在上下文工程的进阶讨论中，来源对比了 **MCP（模型上下文协议）** 与 **CLI（命令行界面）**：

- **MCP 的痛点**：MCP 的工具元数据（如名称、参数 Schema）非常庞大，多工具接入时会占用数万 Token，产生高昂成本。此外，每一步工具调用都要回到模型进行调度，导致链路过长。
- **CLI 的优势**：CLI 可以将复杂的任务组装成一行命令执行，无需模型在中途反复干预。这被认为是一种更高效的上下文利用方式，因为它减少了模型与环境交互的轮次，让上下文更聚焦于最终目标的达成。

总结而言，上下文工程是**智能体从“实验性 Demo”走向“工业化生产力”的必经之路**。它通过存储、筛选、压缩和隔离等手段，在有限的 Token 预算内，为模型构建了一个最有利于决策的“人造环境”。

## 上下文（Context）

在 AI 和大语言模型的语境下，**上下文（Context）** 指的是**大模型每次处理任务时所接收到的全部输入信息总和**。

你可以从以下几个维度来深入理解上下文：

### 1. 上下文的具体内容

上下文并不只是你发给 AI 的最后一个问题，它通常包含以下信息：

- **用户的问题**：当前直接输入的指令。
- **历史对话记录**：为了让 AI “记住”之前聊过的内容，程序会自动把之前的对话内容重新发给模型，这些历史记录构成了上下文的一部分。
- **背景资料与相关数据**：通过 RAG（检索增强生成）等技术从外部文档中提取的相关片段。
- **工具信息与执行结果**：AI 可以使用的工具列表，以及工具运行后返回的数据。
- **系统提示词（System Prompt）**：开发者预设的角色设定、做事规矩和输出格式要求。

### 2. 上下文的本质：AI 的“临时记忆体”

大模型本质上是一个数学函数，它本身并不像人类那样具有持久的生物记忆。**上下文就像是 AI 的一个临时记忆体**。通过将历史信息、规则和外部资料全部塞进这个输入框，AI 才能在处理当前任务时具备必要的“背景知识”，从而给出连贯、准确的回答。

### 3. 上下文窗口（Context Window）

上下文是有大小限制的，这个限制被称为**上下文窗口**。

- **容量衡量**：它的大小通常以 **Token**（大模型处理文本的最小单位）的数量来表示。例如，Gemini 2.5 Pro 的窗口可达 100 万个 Token。
- **溢出效应**：一旦输入的信息超过了这个限制，模型就会丢弃最早的内容，只保留最近的信息。这也就是为什么有些 AI 聊得太久后会“忘记”开头聊过的话。

### 4. 上下文工程（Context Engineering）

由于上下文窗口有限且输入信息越多成本越高，因此诞生了**上下文工程**技术。它通过**保存、选择、压缩和隔离**等手段，精细化设计模型的输入内容，旨在让模型在有限的窗口内理解得更准、回答得更好、且消耗的 Token 更少。

总结来说，**上下文就是 AI 决策的依据和环境**。理解了上下文，就理解了 AI 如何在没有真实记忆的情况下展现出“思考”和“协作”能力的底层逻辑。
