在“Agent 核心原理”的背景下，来源将 **React 模式**视为多智能体系统中最基础、应用最广泛的运行范式。它不仅规定了智能体如何将“思考”与“行动”结合，也是理解现代 AI Agent（如 Cline, Claude Code）底层逻辑的关键。

以下是来源对 React 模式的深度看法：

### 1. 核心定义：智能体的“思考与行动”

- **全称与起源**：React 是 **Reasoning and Acting**（思考与行动）的缩写，最初由 2022 年 10 月的一篇论文提出。
- **运作机制**：React 模式将智能体解决问题的过程拆解为一个持续的循环流程，其核心环节包括：
  1. **Thought（思考）**：模型先对当前任务或观察到的结果进行推理，决定下一步该做什么。
  2. **Action（行动）**：根据思考结果，模型发出调用外部工具（如读取文件、运行搜索）的请求。
  3. **Observation（观察）**：由 Agent 主程序执行工具并获取反馈结果，再喂给模型。
  4. **Final Answer（最终答案）**：重复上述循环，直到模型认为信息足够，输出最终结论。

### 2. 实现原理：由“提示词”驱动而非“训练”

- **非训练依赖**：来源明确指出，React 模式的实现与模型的训练过程关系不大，主要靠**系统提示词（System Prompt）**来实现。
- **“迷你剧本”**：系统提示词相当于给模型安排了一份说明书，强制要求其输出必须包含特定的标签（如 XML 格式的 `<thought>` 和 `<action>`）。如果模型不按此规矩办事，Agent 系统就无法解析其意图并驱动工具。
- **格式的灵活性**：React 是一种**设计思想**，并不限定于某种特定格式。虽然 XML 表达更精确且被 Cline 等工具采用，但纯文本（如 `Thought:` 冒号形式）或 JSON 同样可以实现 React 逻辑。

### 3. 在 Agent 体系中的战略地位

- **单兵作战的基石**：React 是目前构建智能体（如简易版 Claude Code）最常用的模式。它让模型不再只是一个“文字接龙机器”，而是拥有了“感官和四肢”。
- **与其他模式的对比**：
  - **对比 Plan and Execute（规划与执行）**：React 侧重于“边走边看”的动态调整；而 Plan and Execute 则是先生成一整个代办列表，再逐一执行，并根据结果动态修改计划。
  - **协作层级**：在多智能体协议（如 A2A）中，React 通常作用于**智能体内部**的决策流，而 A2A 则负责智能体之间的沟通。

### 4. 现实挑战：上下文过载

- **动态冗余问题**：来源在讨论“上下文工程”时提到，由于 React 涉及多次“思考-行动-观察”的往复，产生的日志和中间过程会迅速填满模型的上下文窗口（Context Window）。
- **“记性”与“智商”的博弈**：当 React 循环过长时，Agent 会因为“记忆过载”而变得缓慢甚至变笨。因此，必须配合**上下文压缩（Compact）**或**隔离（Isolate）**等工程手段，才能让 React 模式在处理大型复杂项目时保持高效。

总结而言，**React 模式被视为 Agent 的灵魂**。它通过结构化的 Prompt 规约，将原本静态的大模型转变为能够与外部环境实时交互、具有自主规划能力的动态智能程序。
