在更大的“AI 协议规范”背景下，来源将 **Agent Skill（智能体技能）** 视为 AI 领域从单一产品功能演变为**通用工业标准**的关键环节。它与 MCP（模型上下文协议）和 A2A 协议共同构成了多智能体协作与工具使用的规范体系。

以下是来源对 Agent Skill 的核心看法及其在协议规范范畴下的讨论：

### 1. 核心定位：AI 的“行为准则”与“说明文档”

- **定义的标准化**：Agent Skill 被描述为大模型可以随时翻阅的**说明文档**（通常为 `skill.md` 格式）。它不再是硬编码在系统提示词中的一段话，而是一种**动态加载的 Prompt** 规范。
- **从私有到开放**：Anthropic 最初推出 Agent Skill 仅为提升 Claude 的表现，但在 2025 年 12 月 18 日正式将其发布为**开放标准**。目前，Cursor、VS Code 和 Codex 等主流开发工具均已跟进支持，使其成为了 AI Agent 领域的通用设计模式。

### 2. 协议分工：MCP 提供数据，Skill 提供教导

在 AI 协议规范的宏观视角下，来源明确了 Skill 与 MCP 的边界：

- **本质区别**：**MCP 负责“连接”**（连接外部世界与数据，如查询天气或订单状态），而 **Agent Skill 负责“教导”**（教模型如何处理这些数据，如规定会议总结的格式）。
- **协作模式**：MCP 像是一套统一的“Type-C”硬件接口标准，而 Agent Skill 则是运行在上面的“软件规约”或“做事规矩”。在实际场景中，两者通常结合使用：Skill 规定步骤，MCP 执行具体的环境操作。

### 3. 核心机制：渐进式披露（Token 经济学）

来源高度评价了 Agent Skill 在**上下文管理（Context Engineering）**方面的工程智慧，这被认为是 AI 协议进化的方向：

- **三层加载机制**：
  1. **元数据层**：始终可见，包含名称和描述，充当模型的“目录”。
  2. **指令层**：只有当模型识别出需求与该 Skill 匹配时，才会按需加载详细说明，极大地节省了 Token。
  3. **资源层（Reference & Script）**：包含更深层的参考资料或自动化脚本。例如，只有在提到“钱”时，才会加载 `reference` 中的财务手册。
- **脚本执行的优势**：Skill 中的 `script`（如 Python 脚本）只会被执行而不会被模型阅读，这意味着即使脚本有一万行，其消耗的上下文 Token 也几乎为零。

### 4. 架构差异：Skill 与 SubAgent 的上下文隔离

在多智能体规范中，来源对比了 Skill 与 SubAgent 的不同处理路径：

- **上下文共享**：Agent Skill 运行时完全**继承并共享**当前主对话的上下文。每一步思考和日志都会记录在主窗口，适合与当前任务关联度高的操作。
- **上下文隔离**：SubAgent 则拥有**独立**的上下文窗口。它在处理完几万行代码的审核任务后，仅将最终结果汇报给主 Agent，从而保持主对话的干净。

### 5. 行业意义与评价

- **灵活性与安全性平衡**：相比于直接生成 Bash 命令行（CLI），Agent Skill 在提供灵活性的同时，通过结构化的指令约束了模型的行为，使其比纯命令行更安全可控。
- **工业化趋势**：来源认为，通过将 Skill、MCP 和 SubAgent 等能力打包成 **Plugin（插件）**，AI 正在形成类似 macOS 的 DMG 或 Windows 的 EXE 这种“一键安装、即刻获得能力”的工业化生态。

总结来说，在 AI 协议规范的版图中，**Agent Skill 是负责“定义逻辑与规矩”的标准化协议**，它解决了模型在处理海量工具和复杂规则时面临的 Token 过载与指令模糊问题，是实现 AI 真正“工业化生产力”的重要基石。

## Agent Skill（智能体技能）与Prompt（提示词）

 问：Agent Skill ，本质上就是提示词，对吗？
从某种程度上说，你的理解是对的，但并不完全准确。**Agent Skill（智能体技能）在最通俗的层面可以被理解为一种“动态加载的 Prompt（提示词）”**，但它在工程实现和功能上比普通提示词要强大且复杂得多。

根据来源，你可以从以下几个维度来深入理解 Agent Skill 的本质：

### 1. 本质定位：AI 的“说明书”

来源多次提到，Agent Skill 的最通俗定义就是**一份大模型可以随时翻阅的“说明文档”或“说明书”**。它主要用来规定模型在处理特定任务时的做事步骤、规则和输出格式。例如，你可以写一个 Skill 规定模型在做会议总结时必须包含参会人员、议题和决定这三项内容。

### 2. 为什么它不只是“提示词”？

虽然它表现为提示词，但它是一套**标准化的工业协议**（由 Anthropic 发布并成为开放标准），具有普通提示词没有的几个核心特性：

- **渐进式披露（按需加载）**：这是它最核心的机制。如果你直接把几万字的规则塞进系统提示词，会消耗大量 Token 且让模型变傻。Agent Skill 将内容分为三层：
  1. **元数据层**：模型只在“目录”里看到名称和描述。
  2. **指令层**：只有当模型判断需要该技能时，才会加载具体的指令内容，从而**极大地节省 Token**。
  3. **资源层（Reference & Script）**：只有在触发表单特定条件时，才会加载深度参考资料（如财务手册）或运行脚本。
- **能够“跑代码”而不消耗上下文**：Skill 包含 **Script（脚本）** 能力。这些脚本（如 Python）只会被执行，而不会被模型阅读。这意味着即便脚本有一万行逻辑，它消耗的上下文 Token 也几乎为零。
- **上下文隔离**：在某些实现中（如 SubAgent），技能可以拥有独立的上下文窗口，处理完海量信息后只汇报最终结果，保持主对话的整洁。

### 3. Agent Skill 与 MCP 的分工

要理解 Skill 的本质，必须看它与 MCP 的区别。官方有一句核心观点：**“MCP connects data; Skills teach what to do with that data.”**（MCP 负责连接数据，Skill 负责教模型如何处理这些数据）。

- **MCP**：像是一套硬件接口（如 Type-C），让模型能查天气、读数据库。
- **Agent Skill**：像是软件规约，规定模型拿到天气数据后应该如何根据用户的出勤习惯给出建议。

**总结来说：** Agent Skill 本质上是**一份结构化的、支持动态加载和脚本执行的标准化指令文档**。它解决了传统 Prompt 在面对复杂规则和海量工具时容易造成的“Token 过载”和“指令模糊”问题。
