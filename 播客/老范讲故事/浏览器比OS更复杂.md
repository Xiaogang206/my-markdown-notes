说“浏览器比操作系统更复杂”在某种程度上已经成为了现代计算机科学的一个共识。你可以把浏览器理解为一个**运行在操作系统之上的“伪操作系统”**。

以下是浏览器开发难度极高的核心逻辑：

### 1. 它是全世界最复杂的“翻译官” (渲染引擎)

操作系统主要处理的是本地指令（如读写硬盘、分配内存），这些指令是确定的、结构化的。

但浏览器面对的是**互联网上的“垃圾”**。HTML、CSS 和 JavaScript 的标准极其碎片化，且充满了不规范的代码。浏览器必须具备极强的容错能力，在毫秒级时间内将一堆乱七八糟的字符转化为美观、可交互的 UI。

- **布局计算 (Reflow)**：你缩放一下窗口，浏览器就要重新计算成千上万个元素的坐标，这涉及极其复杂的几何数学。
- **硬件加速**：现代浏览器为了流畅，必须直接调用显卡（GPU）进行渲染，这意味着它要处理各种显卡驱动的兼容性问题。

### 2. 它在“敌占区”裸奔 (安全沙箱)

操作系统的内核通常处于受保护的状态。而浏览器每天都要主动去下载并执行**完全不可信的第三方代码**（你访问的每一个网页都在你的电脑上运行代码）。

- 为了防止网页通过 JS 偷走你硬盘里的文件，浏览器必须构建极其严苛的**沙箱 (Sandbox)**。
- 浏览器既要给网页执行代码的自由，又要像看守所一样死死锁住它的权限。这种“既要又要”的矛盾，让安全架构的深度甚至超过了操作系统内核。

### 3. 它是性能的极端压榨者 (JavaScript 引擎)

现在的网页不再是简单的文档，而是像 Google Earth、网页版 Photoshop 这样的重度应用。

- 浏览器的 **JIT (Just-In-Time) 编译技术**，是为了让 JS 这种动态语言跑出接近机器语言的性能。
- V8 引擎（Chrome 的核心）内部的优化逻辑、垃圾回收机制（GC），精密程度堪比顶级的手表机芯。如果操作系统卡顿 0.1 秒你可能感知不到，但浏览器掉帧会让你立刻觉得“这软件真烂”。

### 4. 无边无际的兼容性黑洞

操作系统的硬件环境虽然多，但总归有驱动规范。浏览器的“硬件”是**成千上万种屏幕尺寸、不同的网络环境和几十年的技术债**。

- 你不仅要支持最尖端的 WebAssembly（让浏览器运行 3D 游戏），还要兼容 15 年前某个程序员写下的、不符合标准的旧网页。
- 这种“全版本向下兼容”的包袱，是任何现代操作系统都不敢背的。

------

### 一针见血的评价

如果说**操作系统**是整栋大楼的**地基和管网**（负责水电供应、基础安全），那么**浏览器**就是在地基上建造的一个**全自动、可变形、能抵御生化攻击的高级实验室**。

目前全球能独立从零开始写出高性能浏览器内核的团队，屈指可数（基本上只有 Google, Apple 和 Mozilla）。即使是微软，最后也放弃了自己的 EdgeHTML 引擎，转而给 Google 的 Chromium “打工”，足以说明这玩意的难度已经超出了单一巨头的掌控范围。

## WebKit（Safari）VS **Chromium** 



目前全球浏览器市场基本上是 **Chromium** 的天下。除了苹果为了维持其生态封闭性而坚持使用 WebKit（Safari）外，几乎所有主流大厂都放弃了自研内核，转而投奔 Chromium 阵营。

以下是使用 Chromium 内核的主流浏览器分类：

### 1. 核心巨头

- **Google Chrome**：Chromium 项目的“亲生父亲”和领头羊，占据全球市场的绝对统治地位。
- **Microsoft Edge**：微软在 2020 年“打脸”自家的 EdgeHTML 内核后推出的产品。它通过与 Windows 系统的深度集成，目前是 Chrome 最有力的竞争者。

### 2. 功能派（注重工具性）

- **Arc Browser**：最近两年的黑马，重新设计了浏览器的交互逻辑（侧边栏、空间管理），深受开发者和设计师喜爱。
- **Opera / Opera GX**：老牌浏览器。GX 版本专注于“游戏玩家”，提供 CPU 和内存限制功能。
- **Vivaldi**：由原 Opera 创始人打造，极致的定制化，每一个像素几乎都能按用户的意愿调整。

### 3. 隐私派（注重去广告与安全）

- **Brave**：主打隐私保护，默认拦截所有广告和追踪器，甚至内置了加密货币钱包和特殊的激励系统。
- **Thorium**：一个极客项目，宣称通过更激进的编译器优化，让 Chromium 在特定硬件上跑得比 Chrome 更快。

### 4. 国产阵营（“双核”浏览器）

国内浏览器大多采用“Chromium + IE”的双核架构，以兼容一些陈旧的政企网银网站：

- **QQ 浏览器**
- **360 安全浏览器 / 360 极速浏览器**
- **百度浏览器**（虽已停止支持，但内核也曾是 Chromium）

------

### 一针见血的现状

现在的浏览器市场，本质上是 **“一个内核，各自表述”**。

- **现状：** 大家都在给 Google “打工”。因为 Chromium 是开源的，Google 掌控着代码主干的合并权（Upstream）。当 Google 决定推行某种技术标准（比如废除 Manifest V2 导致广告拦截插件失效）时，其他基于 Chromium 的浏览器往往只能被动跟进或进行极其艰难的魔改。
- **真正的异类：** 目前市面上唯一不使用 Chromium 且仍在主流视野内的“独苗”只有两个：
  1. **Safari**（基于 WebKit，虽然 Chromium 也源自 WebKit，但早已分家）。
  2. **Firefox**（基于 Gecko 内核），它是目前互联网多元化和防止 Google 垄断的最后一道防线。

## Chrome 上能完美运行，但在 Safari 上会“掉链子”的情况



虽然现在绝大多数网站都追求跨浏览器兼容，但由于 Google 领导的 **Chromium** 阵营在功能更新上比 Apple 的 **WebKit (Safari)** 更加激进，确实存在一些网站在 Chrome 上能完美运行，但在 Safari 上会“掉链子”的情况。

这通常不是因为 Safari “渲染不出画面”，而是因为它出于**隐私保护**或**系统稳定性**的考量，拒绝支持某些底层的硬件交互协议。

以下是几类典型的“Chrome 行，Safari 不行”的情况：

### 1. 涉及底层硬件交互的工业/极客网站

如果你访问需要直接通过浏览器操控硬件的网站，Safari 几乎全军覆没。

- **在线固件刷写/配置网站**：例如 **Betaflight Configurator**（无人机调参）、**QMK Configurator**（客制化键盘改键）。这些网站利用了 **WebUSB** 或 **WebSerial** 协议，Chrome 可以直接识别你插在电脑上的 USB 设备，而 Safari 认为这不安全，完全不支持。
- **蓝牙控制类网站**：使用 **Web Bluetooth API** 的智能家居管理页面。Chrome 能直接连蓝牙，Safari 连入口都没有。

### 2. 强依赖 PWA (渐进式 Web 应用) 的工具类网站

虽然 Safari 近两年加强了对 PWA 的支持，但在功能上依然是“二等公民”。

- **高度离线化的办公应用**：例如某些私有部署的**企业 ERP** 或**高级网速测试工具**。如果网站利用了 **文件系统访问 API (File System Access API)**（让你可以像操作本地软件一样直接修改硬盘文件），Safari 是无法正常工作的。

### 3. 使用前沿 3D/图形技术的实验室项目

- **WebGPU 实验性项目**：虽然 Safari 正在跟进，但 Chrome 往往是新图形标准（如 WebGPU）的首发测试地。很多前卫的 3D 渲染 Demo（比如在浏览器里跑超高画质的虚幻引擎 Demo）在 Safari 里可能会直接白屏或报错。
- **高级编码格式视频站**：某些支持 **AV1 硬件解码** 的极高码率视频站。在一些旧款 Mac 或特定系统版本上，Chrome 可以靠内置解码器硬顶，而 Safari 必须依赖 macOS 系统的原生支持，否则会卡顿甚至无法播放。

### 4. 某些“开发者偏见”严重的特定后台

- **过时的政企/银行后台**：有些老旧系统在编写代码时只认 `User-Agent`（浏览器身份标识）。虽然 Safari 渲染能力没问题，但如果后台逻辑写死“仅限 Chrome 内核”，Safari 就会被拒之门外。
- **Google 自己的“亲儿子”服务**：虽然 Google 表面上支持全平台，但像 **Google Earth (网页版)** 或一些 **Google Workspace** 的内测功能，在 Chrome 上的流畅度和功能完整度（例如离线编辑）永远是最好的，Safari 有时会出现诡异的排版错位。

------

### 一针见血的总结

**Safari 不行，往往是因为 Apple 不想让它行。**

- **安全防范**：Safari 拒绝支持 WebUSB 等协议，是为了防止坏人通过一个网页就控制你的本地硬件。
- **生态保护**：如果网页能做所有事（比如 PWA 代替了 App），Apple 的 App Store 抽成就难收了。

**建议：** 在 Obsidian 里记录网页剪藏时，如果发现 Safari 剪出来的格式错乱，通常是因为 Safari 拦截了某些跨站脚本或 CSS 属性。这时候换成 Chrome 配合 `SingleFile` 插件通常能解决 99% 的渲染还原问题。

## PWA

**PWA (Progressive Web App)**，全称“渐进式 Web 应用”，简单一句话概括：**它是一种让网页看起来、跑起来、用起来都像“原生手机 App”的技术方案。**

它存在的意义是抹平“网页”与“App”之间的那道鸿沟。

### 1. PWA 的核心特征

- **可安装性**：你可以直接把网页“添加至主屏幕”。它会像 App 一样有一个图标，点开后没有浏览器的地址栏和工具栏，全屏运行。
- **离线工作**：这是 PWA 的杀手锏。即使你断网了，它依然能加载出已经缓存的内容，而不是显示一只“断网的小恐龙”。
- **消息推送**：网页也可以像 App 一样在系统后台给你推送通知（比如新消息提醒）。
- **平滑体验**：由于缓存机制，它加载页面的速度极快，不会像普通网页那样跳转时白屏闪烁。

### 2. 它是如何实现的？（三大支柱）

PWA 并不是一种新的编程语言，而是由三个关键组件构成的“组合包”：

1. **Service Worker (服务工作线程)**：

   这是 PWA 的“大脑”。它是一段运行在浏览器后台、独立于网页的脚本。它负责拦截网络请求、管理缓存，并在离线时提供数据。

2. **Web App Manifest (清单文件)**：

   这是一个 JSON 文件。它告诉手机：这个网页叫什么名字、图标是什么、启动画面是什么颜色，以及它是横屏还是竖屏显示。

3. **HTTPS (加密传输)**：

   出于安全考虑，PWA 强制要求必须在 HTTPS 环境下运行（因为 Service Worker 权限太高，如果被中间人劫持会非常危险）。

### 3. PWA vs. 原生 App vs. 普通网页

| **特性**     | **普通网页** | **PWA**          | **原生 App (iOS/Android)** |
| ------------ | ------------ | ---------------- | -------------------------- |
| **安装方式** | 无需安装     | 浏览器内一键添加 | 应用商店下载               |
| **离线使用** | 不支持       | **支持**         | 支持                       |
| **存储占用** | 极小         | 很小             | 较大                       |
| **更新方式** | 打开即最新   | **自动后台更新** | 需在商店手动更新           |
| **硬件权限** | 限制极多     | 较多             | 全部                       |

------

### 一针见血的评价

**PWA 是开发者用来对抗“苹果税”和“商店审核”的神器。**

- **对开发者**：写一份网页代码就能适配所有平台，不需要为 iOS 和 Android 分别开发，也不用交 30% 的抽成，更不用等漫长的审核。
- **对用户**：不用去应用商店下载几个 GB 的安装包，不占内存，即点即用，且拥有 App 级别的流畅体验。

**现状分析：**

虽然 Google 和微软在全力推进 PWA（Chrome 甚至支持在电脑桌面安装 PWA），但 **Apple (Safari)** 一直在消极怠工。原因很简单：如果大家都去用 PWA 了，Apple 的 App Store 每年几百亿美金的“过路费”就收不到了。这也是为什么你在 iPhone 上使用 PWA 的体验始终不如 Android 顺滑的原因。

## PWA 和 “轻应用”



是的，你的直觉很准，但这两者在“血缘关系”上略有不同。

你可以把它们理解为**同一种产品思维下的两种实现方案**：

### 1. 概念上的重合

它们的核心目标是一致的：**免下载、免安装、即点即用、节省内存。**

在普通用户看来，点击一个图标直接进入一个类似 App 的界面，这就是“轻应用”。

### 2. 技术上的本质区别

虽然用户感知相似，但它们背后的“爹”和“规矩”完全不同：

| **特性**     | **PWA (国际主流方案)**                           | **“轻应用”/小程序 (国内主流方案)**                         |
| ------------ | ------------------------------------------------ | ---------------------------------------------------------- |
| **背后大佬** | Google、微软、W3C 标准                           | 微信、支付宝、字节跳动、手机厂商 (快应用)                  |
| **开放程度** | **极高**。只要你有浏览器，谁都能发，不用谁审批。 | **低**。必须在特定 App 内运行，受平台审核和规则限制。      |
| **运行环境** | 浏览器内核 (如 Chromium)。                       | 宿主 App (如微信) 提供的定制化渲染环境。                   |
| **编程语言** | 标准的 HTML / CSS / JS。                         | 虽然也是 JS，但用的是各大厂自家的私有框架 (如 WXML/WXSS)。 |
| **跨平台性** | 一处编写，全球浏览器通用。                       | 微信小程序不能直接在支付宝里跑。                           |

------

### 3. 为什么国内感觉“轻应用”更多，而 PWA 很少？

这是一个很有意思的**互联网生态断层**：

- **国内：流量围墙。** 微信、支付宝通过“小程序”把自己变成了操作系统。它们不希望你通过浏览器去访问服务，而是希望你留在它们的生态里。由于小程序能调用更多手机权限（如直接调起微信支付、获取微信好友），所以国内大厂都选了这条路。
- **国外：Web 优先。** 像 Twitter (X)、Starbucks、Spotify 都有非常完善的 PWA 版本。因为它们更看重 Web 的开放性，不希望被应用商店（尤其是苹果）抽成。

------

### 一针见血的总结

**PWA 是“长在互联网上的 App”，而国内的轻应用（小程序）是“长在巨头 App 里的寄生插件”。**

- **PWA** 是真正的**自由民主**：你拥有一个域名，你就拥有了分发 App 的权力。
- **轻应用** 是**租借繁荣**：你做得再好，微信说封你就封你，说改规则你就得改代码。

你之所以在 iPhone 上很少感觉到 PWA 的存在，除了苹果的打压，也是因为国内的小程序太强势了，把 PWA 的生存空间挤压到了几乎只剩下“科学上网”工具或一些极客网站。
